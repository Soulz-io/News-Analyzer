## TECHNICAL CONTEXT

### API Dispatch Map
The frontend calls APIs via `?_api=X` query params on the root URL. The dispatch map is in `engine/engine.py` starting at line ~1545 (`_API_DISPATCH` dict). Key mappings:
```python
"overview": "/api/dashboard/overview",
"status": "/api/status",
"signals": "/api/signals",
"indicators": "/api/indicators",
"briefs": "/api/briefs",
"swarm-feed": "/api/swarm/feed",
"swarm-status": "/api/swarm/status",
"advisory": "/api/advisory/latest",
"advisory-history": "/api/advisory/history",
"usage-breakdown": "/api/usage/breakdown",
"admin-users": "/api/admin/users",
"tree": uses _API_PARAM_DISPATCH with id parameter
```

### Swarm Consensus Flow
1. Scheduler triggers `swarm_consensus_job()` every 60 minutes
2. Finds pending decision nodes to evaluate
3. For each node: 12 experts × 2 rounds = 25 LLM calls
4. Round 1: Independent assessment
5. Round 2: Debate (experts see each other's Round 1 answers)
6. Synthesis: Final verdict (STRONG_BUY/BUY/HOLD/SELL/STRONG_SELL)
7. Stored in `swarm_verdicts` table
8. After swarm: triggers `generate_refresh_advisory()` if significant changes

### Database Tables (key ones)
- `users` — id, username, email, password_hash, portfolio_json, is_admin, onboarded
- `articles` — raw RSS/GDELT articles
- `article_briefs` — NLP-enriched articles (sentiment, intensity, keywords)
- `narrative_timelines` — daily narrative tracking
- `run_ups` — detected escalations
- `decision_nodes` — game theory decision trees
- `consequences` — outcomes per tree branch
- `swarm_verdicts` — expert panel verdicts (round1_json, round2_json, entry_reasoning, dissent_note)
- `trading_signals` — composite confidence scores
- `analysis_reports` — daily advisories (report_json)
- `engine_settings` — key-value config store

## APPROACH RECOMMENDATIONS

### For UI changes (app.js)
- Multiple agents writing to app.js WILL create conflicts
- **Better approach**: Have ONE agent do all app.js changes, or split by clear function boundaries
- Each function in app.js is self-contained — you can replace entire functions safely

### For backend changes (api_routes.py, swarm_consensus.py)
- These are safer to parallelize since different functions don't overlap
- Always run `python3 -c "import py_compile; py_compile.compile('file.py', doraise=True)"` after changes

### For new experts (swarm_consensus.py)
- Experts are defined as dicts in a list (lines 102-345)
- Each expert has: role, system_prompt, provider (groq/openrouter), model
- Adding 30 experts doesn't mean they all run — add an `enabled: bool` flag
- Budget impact: each additional expert = +2 LLM calls per evaluation cycle
